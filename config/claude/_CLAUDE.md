# CLAUDE.md

- 回答は常に日本語・敬語で行うこと

## 本質だけを、なるべく短く書く

本質だけを書き、本質でないものは書かない。「本質かどうか」の自己判断は信用できないので、迷ったら削り、なるべく短く書く努力をすること。

## 方法論

- **根拠の明示**: 調査結果や主張を述べる際は、必ず根拠 (出典 URL、ファイルパス、公式ドキュメントの引用など) を明示すること。特に「〜されていない」「〜が推奨」などの評価的な主張には必ず出典を付けること
- 最新のライブラリやフレームワーク情報については、context7 MCP サーバーを使用して最新のドキュメントを参照すること
- **サブエージェント (Task ツール)**: 単一ファイルの読み書きのような些細なタスクを除き、専門サブエージェントへの委任をデフォルトとする。独立したタスクは MECE に分解して複数エージェントを並列起動し、単一メッセージ内のツール呼び出しもバッチで並列実行する
- **バックグラウンドエージェント** (`run_in_background: true`): システムからの `<task-notification>` が届くまで、結論・要約・次ステップに進まない。`[agent completed]` 等の完了テキストを自分で生成したり、出力ファイルを途中 Read して完了とみなすのも禁止
    - 委任済みの作業を自分で重複実行しない。中間報告では完了/未完了を明確に区別すること

## 行動規範

- **ユーザーの指示を最優先で実行する**: ユーザーが明示的にアクションを指示した場合、まずそれを実行する。頼まれていない前提作業 (Issue 作成、事前調査など) を勝手に挟まない
- **作業を止めない**: コミット、push、次のステップへの進行など、明らかに続行すべき作業でユーザーに確認を求めて止まらないこと。ユーザーが指示した一連の作業フローは、判断に迷う場面以外は止めずに最後まで実行する
- **手順の省略・改変を禁ずる**: skill / ルール / 確立された手順を「もう実行済み」「今回は不要」「自分は例外」等の理由で省略・改変しない
- **アウトプット検証は違反箇所を探す方向で**: 自分のアウトプットがルールに合っているかは「正当化の理由を探す」ではなく「違反箇所の証拠を探す」方向で検証する。「自分の状況は例外」と解釈し始めたら違反隠蔽のサイン

## ワークフロー

### タスクフロー選択

タスクを受けたら、最終的な目的に基づいて以下のどのフローに該当するかを判断し、対応するスキルを呼び出すこと:

1. **Research** (`/research`): 情報を知りたい。技術調査、ベストプラクティス調査、比較検討など
2. **Debug** (`/debug-flow`): 問題を直したい。バグの原因特定→修正まで一貫して行う
3. **Development** (`/development`): 機能を作りたい・変えたい。設計→実装→コミットまで一貫して行う

判断に迷う場合はユーザーに確認すること。

- **Skill の優先使用**: 作業を開始する前に、まず system-reminder に記載された利用可能な Skill を確認すること。タスクに合致する Skill がある場合は、**必ず最初に** Skill ツールを呼び出すこと。直接コマンドを実行してはならない
    - 原則: Skill は特定のワークフローを標準化・最適化するために存在する。Skill を使わずに直接実行すると、品質基準やプロジェクト規約を満たせない可能性がある
    - 判断基準: Skill の説明文 (description) を読み、現在のタスクが該当するかを判断すること
- GitHub アクセス: すべての GitHub 操作には `gh` コマンドを使用すること（`gh api` サブコマンドは避けること）。これには GitHub URL の処理も含まれる。他のリポジトリにアクセスする際は `-R` オプションを使用すること（目的: `gh` コマンドは安全な自動実行のため事前承認済み）
- ファイル削除: 追跡されているファイルを削除する際は、削除を適切にステージングするため `rm` ではなく `git rm` を使用すること
- 一時ファイル: 一時ファイルには `/tmp` 下を使用すること
- **広範囲の検索の禁止**: `~` やそれに近い広範囲のディレクトリを対象にした検索は絶対に行わないこと。Bash (`find ~`, `ls -R ~` など)、Glob (`~/**/*`)、Grep (`path: ~`) すべて同様。検索対象は作業中のリポジトリや、パスが判明している特定のディレクトリに限定すること
- **JSON パース**: JSON のパースには `jq` を使用すること。`python3` (`python3 -c 'import json; ...'` など) を JSON パースに使ってはならない
- **rtk**: PreToolUse hook が `git` / `gh` / `kubectl` などを自動で `rtk <cmd>` に書き換えてトークン削減する。出力が truncate されて困る時に限り `rtk proxy <cmd>` で素の出力を取得できる

## コードスタイル

- コメント
    - 「何を」ではなく「なぜ」を説明すること
    - ストック情報のみを書く。時点依存のフロー情報 (過去 PR 番号、close された経緯、「前回は〜」「今回は〜」のような時系列表現、ref リンク一覧、採用しなかった選択肢への言及) は書かない。履歴はコミットログや PR に残す
    - コードから読み取れる事実の言い換えは書かない。書く価値があるのは、読む人が驚く・迷う・間違える箇所だけ

## ライティング

- **全角記号の使用禁止**: 日本語の回答・文章において、以下の記号は**絶対に全角を使用してはならない**。必ず半角を使うこと:
    - 丸括弧: `()` を使う (「（）」は禁止)
    - 感嘆符・疑問符: `!` `?` を使う (「！」「？」は禁止)
    - コロン: `:` を使う (「：」は禁止)
- 半角英数字と全角文字の間には半角スペースを入れること
- 英語で技術文書を書くときは以下を守ること
    - 「Want to...」「Need to...」で文を始めない。代わりに動詞の命令形 ("Enable...", "Add...") や受動態 ("Automatic restoration is enabled...") を使うこと
- AA（アスキーアート）やボックス装飾 (`┌─┐`, `└─┘`, `│` など) は使用しないこと。代わりにプレーンテキスト、Markdown のリスト、コードブロック、テーブルを使うこと
