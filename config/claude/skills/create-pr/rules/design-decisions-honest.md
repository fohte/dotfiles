# Design decisions は実際に検討した案だけを書く

Design decisions の「却下」行には、**自分が本当に検討して比較した上で却下した案** だけを書く。テーブルを埋めるためにそれっぽい案を後付けで足してはいけない。

特に次のものを「却下案」として並べてはいけない:

- タスク指示や既存ルールで **最初から禁止されていた anti-pattern** (検討の余地なく避けたもの)
- 採用案を引き立てるためにでっち上げた、誰もまともに検討していない比較対象

これらを「却下」として並べるのは、設計判断を捏造して水増ししているのと同じ。レビュアーは「比較した結果こうなった」という前提で読むので、捏造があると判断の信頼性が崩れる。

判定基準: その「却下案」を採用案と同列に並べたとき、自分は本当にその選択肢を採るか迷ったか? 迷っていない (最初から無いと分かっていた) なら書かない。

書いた却下案がない判断軸であれば、テーブルにせず採用案を箇条書きで述べるか、判断軸ごとセクションを削る。比較表という形式に合わせるために案を増やす必要はない。

## 判断軸そのものを後付けで作らない

「却下案を捏造するな」と同じ理由で、**判断軸そのものを後付けで増やすのも禁止**。比較表を厚く見せるために、実際には迷っていない決定を「判断軸」として取り出して並べないこと。

判定基準: その判断軸の採用理由を書こうとしたとき、対立する代替案が自然に出てこない (=「これ以外を選ぶ理由がない」と感じる) なら、それは判断軸ではなく単に採用案の説明。Design decisions に並べず、Approach の本文に 1 行で書くか、何も書かずに済ませる。

特に、表現スタイル・例示の見せ方・命名・コメントの書き方など「センスの問題」に近い決定は判断軸として立てない。これらは比較対象を作ろうとすれば作れてしまうため、捏造の温床になりやすい。

- ❌ 「bad 例として実コードを引用するか / 擬似コードを書くか」を判断軸として立てる (どちらかを真剣に検討したわけではないのに、表を埋めるために対立構造に整形している)
- ❌ 「変数名に full word を使うか / 略語を使うか」を判断軸として立てる (慣例に従っただけで比較していない)
- ✅ 採用案の理由が単独で述べられるなら、判断軸セクションを作らず Approach に 1 行書く
