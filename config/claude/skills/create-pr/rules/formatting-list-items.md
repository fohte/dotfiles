# 箇条書き項目のフォーマット

## 補足は括弧書きで流さずサブバレットにする

親項目に対する補足・但し書き・例外条件・限定条件は、`(...)` の括弧書きで親項目の文末に流さず、サブバレット (インデントした箇条書き) として書く。括弧書きで詰め込むと親項目の主張と補足の関係 (主張 → 例外、主張 → 根拠) が読み解きにくく、レビュアーに余計な負担をかける。

判定基準: 親項目の文末に `(...)` で何かを書こうとしたら、その内容を一段下のサブバレットに切り出せるか試す。切り出して読みやすくなるならサブバレットにする。

短い言い換え・単位・略称展開のような「親項目の語句の補強」は括弧書きで構わない (例: `単位 (= ms) を揃える`、`タイムアウトを 30 秒 (= 30000 ms) に設定する`)。

### ❌ 括弧書きで補足を流す

```markdown
- 入力をブロック単位で評価する (構文上の特殊ケースは引き続き 1 ブロック扱いのまま保持する)
- 失敗時はエラー終了する (旧来のフォールバック挙動を廃止)
```

### ✅ サブバレットに切り出す

```markdown
- 入力をブロック単位で評価する
    - 構文上の特殊ケースは引き続き 1 ブロック扱いのまま保持する
- 失敗時はエラー終了する
    - 旧来のフォールバック挙動を廃止
```

## 項目の中に空行を挟まない

箇条書きの 1 項目の中に空行を入れない。空行を入れると Markdown レンダリング時にその項目が "loose list" 扱いとなり、各項目が `<p>` タグで囲まれる。結果として項目同士の間隔が広がり、箇条書きとしての一覧性が失われる。

子要素 (引用ブロック・コードブロック・サブバレット) を続ける場合は、親項目の直後に空行なしでインデントして置く。

### ❌ 空行を挟む

```markdown
- 親項目の説明

    > 引用ブロック
    > [出典](URL)

- 次の項目
```

### ✅ 空行を挟まない

```markdown
- 親項目の説明
    > 引用ブロック
    > [出典](URL)
- 次の項目
```

### コードブロックの場合も同様

````markdown
- 親項目の説明
    ```yaml
    key: value
    ```
- 次の項目
````

### 段落を分けたいときは

「同じ項目の中で段落を分けたい」と感じたら、それは項目を分ける合図。別の箇条書き項目に切り出すか、項目の外に段落として書く。
