# What は Why の回答になるように書く

Why で述べた問題・目的に対して、この PR がどう解決/実現するかを説明する。

- Why: 「並列実行時にセッション状態が把握しにくい」→ What: 「セッションの状態と所在を一覧で確認できるようにする」
- ❌ いきなりコマンドや機能を羅列しない (Why との繋がりが見えない)

技術的原因は What の子要素に書く。バグ修正の場合、What の 1 行目に解決策を書き、子要素として技術的原因を補足する。

```markdown
## What

- 範囲指定子を含むバージョンを正しく抽出・比較できるようにする
    - 正規表現が `v` プレフィックスのみを考慮しており...
    - YAML の unquoted version 比較で `^` が...
```

## 子要素は現状の仕組みの説明だけにする

What に書いてよい子要素は、対象のコード/状態が**現在どういう仕組みで動いている/壊れているか**の説明だけ。「どういう経緯で今の状態になったか」は実装方針と無関係なので書かない。

判定基準: 子要素を読んだレビュアーが「だから 1 行目の対応が要るのか」と納得できるなら残す。「だから何?」となるなら経緯の説明であり、削る。

- ✅ 現在の挙動・構造の説明 (1 行目の解決策が必要な理由になっている)
- ❌ 過去の出来事の順序や、その時の状況の説明 (現状の仕組みではなく履歴)
