# 変更の種類に応じて書き分ける

- **バグ修正・改善の場合**: 「何が問題か」(症状/影響) を最初の箇条書きで述べる
- **新規追加の場合**: 「何を実現したいか」(目的/動機) を最初の箇条書きで述べる

## 新規追加では「機能の説明」だけで終わらせない

新規追加で機能/リソースの一行説明だけを書いても、レビュアーは「なぜ今これを作るのか」が分からない。実現したい機能名だけでなく、その背後にある**動機 (どういう用途のために必要になったのか、何がきっかけで始めたのか)** を Purpose に含めること。

判定基準: その Purpose を読んだレビュアーが「なるほど、だからこの PR を出したのか」と納得できるか。「で、なんでこれが今必要なの?」が残るなら動機が抜けている。

- ❌ `〇〇 リポジトリを Terraform で宣言的に管理できるようにする` (機能の説明だけ。なぜ今これを作るのかが不明)
- ✅ `△△ の受け口として使う 〇〇 リポジトリを新たに用意したい` (用途・動機が示されている)

委任タスクや issue の「背景」「目的・モチベーション」「直前の経緯」セクションは、まさにこの動機を伝えるために書かれている。**その情報を Purpose に転記しないまま「機能の説明」だけにしてしまうのが最も多い失敗パターン**。タスク指示にこれらのセクションがあれば、必ず Purpose に反映させること。
