# PR タイトルと description の書き方ガイド

ドラフト作成時は以下のルールに従うこと。各ルールの詳細と具体例は `rules/<ルール名>.md` を参照。

**重要:** このファイルの表は概要でしかない。ドラフトを書き始める前に、下の Why/What の表に列挙されている `rules/<ルール名>.md` を**全て**読むこと。表の一行説明だけで判断してルール個別ファイルをスキップすると、表現されていない具体例 (❌/✅ パターン) に気づけず、違反したドラフトを書いてしまう。「自分の変更に関係ありそうなものだけ読む」という自己判断は禁止。

## 基本ルール

- **Why/What のみ記述**: それ以外のセクション (「期待される効果」「参考」など) は書かない
- **コード要素はバッククォートで囲む**: ファイル名、関数名、変数名、コマンド名など

## タイトル

| ルール                  | 概要                                 |
| ----------------------- | ------------------------------------ |
| title-objective-summary | 客観的な変更の要約。結果を断定しない |

## 全般: タスク指示と PR description は読者が違う

委任タスクや issue の指示は実装者向けに書かれており、技術的な詳細 (data source 名、具体的な API、検証結果など) を多く含む。PR description はレビュアー向けであり、そのまま転記すると情報過多になる。ドラフトを書く前に「この情報はレビュアーにとって必要か?」を判断し、実装の詳細はすべて削ぎ落とすこと。

特に次のパターンは指示文からの転記になりやすい。該当する文を本文に書いた時点で削ること:

- **「〇〇は残す」「△△は変更しない」という注意書き**: 指示文では実装者が誤って消さないための注意だが、レビュアーは diff を見るので「残したもの」は見えない。言及する必要がない
- **action/image の SHA、バージョン番号、具体的な pin 値**: 指示文には「この pin を使え」と書かれるが、レビュアーは diff で見れば済む
- **「なぜその手順を取るか」の実装上の理由**: 例えば「helm plugins を入れるため〇〇が必要」のような説明は、実装者が手順を飛ばさないための情報であり、diff だけ見るレビュアーには不要

指示文に書かれた情報を本文に書き写そうとしたら、まず「これは実装者向けの注意か、レビュアー向けの変更内容か」を判定すること。前者なら書かない。

## Why セクション

| ルール                  | 概要                                             |
| ----------------------- | ------------------------------------------------ |
| why-symptoms-only       | 技術的原因ではなく症状/影響だけを書く            |
| why-bug-vs-new          | バグ修正は症状から、新規追加は目的から書き始める |
| why-no-condition-start  | 「〜とき」「〜ので」で始めない。主語と結論を先に |
| why-no-lack-description | 「〜がない」ではなく実現したいことを書く         |
| why-cite-with-quote     | 根拠はリンク + 引用文をセットで。from: 表記      |
| why-judgment-reason     | 変更の判断理由を子要素で補足                     |

- **事前に参考リンクの中身を確認する**: タスクの指示、委任タスク、issue に参考リンク (PR URL、issue URL など) が記載されている場合、ドラフトを書き始める前に必ずそのリンク先の内容を読むこと。issue のコメント欄にも検証結果や根拠が含まれていることが多いため、本文だけでなくコメントも確認する
- **Why に解決策の方向性を混ぜない**: 「〇〇が問題。△△なら動くことが確認済み」のように、問題の記述に続けて解決策の示唆を書かない。Why は問題/制約の事実だけに留め、どう解決するかは What に書く
    - ❌ `〜が未対応。〇〇でのユーザー ID 個別指定なら動作することが確認済み` (解決策の方向性が混在)
    - ✅ Why: `〜が未対応` → What: `〇〇を設定するようにする`

## What セクション

| ルール                           | 概要                                                                                                    |
| -------------------------------- | ------------------------------------------------------------------------------------------------------- |
| what-not-how                     | What (何を) を書く。How (どのように) は書かない                                                         |
| what-answers-why                 | Why の回答になるように書く。技術的原因は子要素に                                                        |
| what-no-implementation-breakdown | diff でわかる内訳 (ファイル名、件数、設定キー等) を並べない。タスク指示の技術的詳細の転記も内訳にあたる |
| what-show-examples               | 変更前後の具体例をコードブロックで示す                                                                  |
| what-flexible-format             | 無理に箇条書きにせず、内容に応じた形式を使う                                                            |
