# Issue / コメントの書き方ガイド

このドキュメントは GitHub Issue の本文とコメントの書き方ルールを定義する。

## 基本ルール

- である調を使う (ですます調は使わない)
- 太字 (`**...**`) は使わない。強調が必要なら構造や文言を見直す
- コード要素 (ファイル名、関数名、変数名、コマンド名など) は必ず `` ` `` で囲む
- 推奨や提案は書かない。事実と調査結果のみを書く

## Issue 本文の書き方

- Issue の目的に沿った内容のみ記載する。調査系 Issue なら「何を調査するか」を書き、調査結果は書かない。結果は別コメントに書く
- What セクションには問題の説明だけでなく、解決策の戦略的意図を含める
    - Bad: `config.yaml の 42 行目を削除する`
    - Good: `上流リポジトリの正規設定を使用する`
- 完了条件は作業手順ではなく、達成状態を記述する
    - Bad: `不正なエントリをファイルから削除する`
    - Good: `変更が本番環境にデプロイされている`
- 親 Issue の参照は Why セクション内で `- from <url>` のリスト項目として書く。本文冒頭に `from:` 行を置かない
    - Bad: 本文冒頭に `from: #491, #510`
    - Good: Why セクション内で `- from #510` とリスト項目にし、インデントで詳細を続ける

## タイトルの書き方

- 問題を具体的に表現する。「改善」「修正」のような曖昧な言葉を文脈なしに使わない
- 抽象的かつ簡潔にする。作業の本質を書き、内部用語や具体的な問題名を避ける
    - Bad: `config parser の "foobar バグ" を修正する`
    - Good: `config parser を更新する`
- 複数の問題が同じ根本原因を持つ場合、個別に列挙せず根本原因でまとめる

## セクション見出しの書き方

- 見出しは簡潔にする。結論が短い見出しに収まるなら要約見出しを使う。収まらなければシンプルな見出し (例: `## 原因`) を使い、結論は本文の冒頭に書く
    - Bad: `調査結果`, `症状` (内容が不明)
    - Good: `404 を返していた 2 つの Pod は proxy レイヤーにあった` (要約見出し)、または `## 原因` の直後に結論を一文で書く

## 結論ファーストの原則

- 結論をセクション見出しの直後に一文で書く。読者が知りたいのは答えであり、過程ではない
    - Bad: `## 原因` の後に調査過程を延々と書き、最後に結論
    - Good: `## 原因` の後に `PR #123 が X を導入し、Y が発生した。` を書き、詳細は下に続ける
- 各情報は結論そのものか、結論を裏付けるものでなければならない。「この情報は結論の理解に必要か?」と問い、不要なら書かない
    - 結論を裏付ける情報 (証拠、データ) → トップレベル
    - 結論の導出過程 (メカニズム、タイムライン、プロセス) → `<details>` ブロック
    - リンク先や実行ログから読者が自明に得られる情報 → 省略

## 構造とフォーマット

- トップレベルのテキストは短く保つ。3-4 段落を超える場合は `<details>` ブロックへの移動を検討する
- 結論が読者に馴染みのないシステムやコンポーネントに関するものなら、理解に必要な最小限の背景をトップレベルに書く。詳細なアーキテクチャやデータフローは `<details>` ブロックに入れる
- リンク先で読める情報を繰り返さない。PR や Issue をリンクしたら詳細な再説明は不要。一文の要約で十分
- 箇条書きの各項目を「要約: 説明」の形式で始めない。普通の文で書く
- 順序に意味がある場合 (手順、優先度) のみ番号付きリストを使う。それ以外は箇条書きにする
    - Bad: `1. 選択肢 A 2. 選択肢 B 3. 選択肢 C` (順序に意味がない)
    - Good: `- 選択肢 A - 選択肢 B - 選択肢 C`

## 事実と根拠

- 事実 (エラー頻度、影響範囲など) を述べるときは、読者が検証・再現できるよう証拠を添える (使用したクエリ、ダッシュボードリンク、実行コマンドなど)
- 症状と根本原因を区別する。「X が遅い」「Y にラグがある」は症状であり原因ではない。原因が不確かなら留保表現を使う (例: 「X が原因と推定される」)
- 根本原因と無関係な情報は、調査中に発見したものであっても原因分析や解決策に含めない
- 解決策を提案する場合は、実際に機能することを確認してから書く。例えば「新しいバージョンにアップグレードする」と提案する前に、そのバージョンで問題が修正されているか upstream の Issue、リリースノート、CHANGELOG を確認する

## リンクの書き方

- 参照リンクは文脈の中にインラインで配置する。末尾に「参考」セクションを設けない
- 外部リソースにリンクする際は、そのリンクが関連する理由を説明する
- GitHub ソースコードへのリンクはパーマリンクを使う (commit SHA を含める)
    - Bad: `github.com/org/repo/blob/main/path/file.ts#L10`
    - Good: `github.com/org/repo/blob/abc123.../path/file.ts#L10`
    - 最新 commit SHA の取得: `gh api -X GET repos/org/repo/commits -F path=<file> -F per_page=1 --jq '.[0].sha'`
- 括弧内に URL を置く場合は必ず Markdown リンク構文を使う (GitHub が URL 境界を誤認するのを防ぐ)
    - Bad: `問題が発覚した (https://github.com/org/repo/issues/123)。` -- GitHub が `。` を URL に含める可能性がある
    - Good: `問題が発覚した ([#123](https://github.com/org/repo/issues/123))。`

## 書かないこと

- 「次のアクション」「TODO」「提案する次のステップ」セクションは、ユーザーが明示的に要求しない限り書かない
- コメントは最小限にする。読者が行動するか結果を理解するために必要なことだけ書く。実行ログやリンク先から導ける補足情報は書かない

## リポジトリ慣習の確認

- 書き始める前に、既存の類似 Issue を `view` で確認し、フォーマット (親 Issue リンクの位置、セクション構造など) を合わせる
