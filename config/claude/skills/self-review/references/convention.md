# Convention Review Reference

最初に `references/_common.md` を読み、動作原則・禁止事項・出力形式ボイラープレートを適用する。本ファイルは convention グループ固有の差分のみを記述する。

このグループはプロジェクト固有ルール遵守を担当する。**最初に必ず規約を読み込み、チェックリスト化する** こと。スキップは禁止。

## 担当観点

各観点について「この diff でこの軸は壊れうるか」を必ず一度問い、結論を出力に残す。

7. **可観測性 (Observability)**: ログ・メトリクス・エラー追跡・デバッグ可能性。
8. **ドキュメント整合性**: コード ↔ コメント ↔ doc comment ↔ README ↔ `--help` ↔ 仕様 / 規約 の drift。
9. **コメントの質**: 新規・変更されたコメントを **1 行ずつ** 以下に照らす。1 行でも違反していれば指摘する。
    - 「何を」ではなく「なぜ」を説明している (コードから読み取れる事実の言い換えを書かない)
    - ストック情報のみ (時点依存のフロー情報 — 過去 PR 番号、close された経緯、「前回は〜」「今回は〜」、採用しなかった選択肢への言及など — を書かない。履歴はコミットログや PR に残す)
    - 書き手の判断の正当化・弁明を含まない (「自分が下した設計判断がなぜ正しいか」「この方法でなぜ問題が起きないか」はコミットメッセージや PR 本文の責務)
    - 読む人が驚く・迷う・間違える箇所にだけ存在する (隠れた制約、subtle な invariant、特定バグの workaround、surprising な挙動)
    - **判定基準**: そのコメントを消したら読み手がコードを誤読するか / 重要な制約を見落とすかを問う。「あった方が親切」「書き手の意図が伝わる」程度なら不要
10. **プロジェクト規約遵守**: 上記「規約読み込み」で読んだルールに違反していないか。違反指摘時は **必ず規約ファイルの該当行を引用** する。

## 固有原則・重要度補足

- 規約違反指摘では **必ず** 規約ファイルのパス、行番号、該当文言の引用を併記する。
- 🔴 Critical の典型例: ドキュメントが嘘になっている (古い API を `--help` が説明している等の drift) / 致命的な観測欠如 (失敗が silent)
- 🟡 Warning の典型例: 規約違反 / コメント品質違反 / minor な doc drift
